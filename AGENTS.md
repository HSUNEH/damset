# DamSet agent operating contract

이 파일은 저장소 전체에 적용된다. 에이전트는 코드를 고치는 사람인 동시에
`wiki/`의 유지보수자다. 위키는 대화 기록을 복사하는 곳이 아니라, 다음 작업의
판단 비용을 낮추는 지속 가능한 지식만 축적하는 곳이다.

## 작업 전

1. `git status --short`로 사용자의 기존 변경을 확인하고 보존한다.
2. 반드시 `wiki/index.md`를 읽는다.
3. `rg '^## \[' wiki/log.md | tail -5`와 `tail -80 wiki/log.md`로 최근 작업을
   확인한다.
4. 인덱스의 작업 라우팅 표에서 관련 페이지를 골라 읽는다.
5. 위키만 믿지 말고 현재 코드, 테스트, 설정, 공식 문서를 확인한다. 충돌 시
   현재 실행 결과와 코드가 우선이며, 발견한 차이는 같은 작업에서 위키에
   반영한다.

## 작업 중

- 새로 확인한 내용이 이후 작업의 도구 선택, 설계 판단, 위험 회피, 검증 방식에
  영향을 준다면 관련 위키 페이지를 즉시 수정한다.
- 별도 개념으로 여러 페이지에서 링크할 내용만 새 페이지로 만든다. 기존 주제의
  보완은 기존 페이지를 고친다.
- 사실에는 가능한 한 저장소 상대 경로를 근거로 적는다. 시점에 따라 달라지는
  환경 정보에는 `last_verified` 날짜를 붙인다.
- 페이지 내용을 바꾸면 frontmatter의 `updated`도 작업 날짜로 바꾼다.
- 임시 디버그 출력, 실패한 단발성 시도, 대화 전문, 비밀값, 쉽게 코드에서
  다시 찾을 수 있는 세부 구현은 위키에 쌓지 않는다.
- 코드와 위키가 함께 바뀌어야 하는 작업에서는 둘을 같은 변경으로 다룬다.
  완료 로그는 검증 결과가 나온 뒤 한 번만 추가한다.

## 작업 후

1. 변경 위험에 맞는 검증을 실행한다. 기준은
   `wiki/playbooks/verification.md`를 따른다.
2. `ruby Scripts/check_agent_wiki.rb`와 `git diff --check`를 실행한다.
3. 페이지를 만들거나 역할을 바꿨다면 `wiki/index.md`의 링크와 한 줄 설명을
   갱신한다.
4. 의미 있는 코드·설정·문서·QA 작업은 `wiki/log.md` 끝에 아래 형식으로
   한 항목을 추가한다. 읽기 전용 질의나 사소한 탐색은 지속 지식이 생긴
   경우에만 기록한다. 항목을 추가하면 로그의 `updated`도 바꾼다.
5. 마지막 diff를 읽어 코드, 위키, 로그가 같은 사실을 말하는지 확인한다.

```markdown
## [YYYY-MM-DD] <kind> | <short title>

- Scope: 무엇을 바꿨는가
- Actions: 중요한 도구와 행동 방식
- Decisions: 이후에도 유지할 판단, 없으면 `None`
- Verification: 실행한 명령과 결과
- Wiki: 갱신한 페이지
```

`kind`는 `feature`, `fix`, `refactor`, `qa`, `docs`, `decision`, `maintenance`,
`setup` 중 하나를 쓴다. 상세 규칙과 페이지 스키마는 `wiki/schema.md`에 있다.
