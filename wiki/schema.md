# 위키 운영 스키마

## 목적

이 위키는 에이전트가 다음 작업 전에 읽고, 작업 중 새로운 지속 지식을
통합하고, 작업 후 변경과 검증을 확인하는 컴파일된 지식 계층이다. 채팅이나
로그를 무작정 보존하는 저장소가 아니다.

설계 출발점은 [Karpathy의 LLM Wiki 패턴](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)
(2026-07-19 확인)이다. 이 저장소에서는 별도 raw 수집함 대신 버전 관리되는
코드·테스트·문서를 근거 계층으로 쓰고, 일반 지식 위키 대신 앱 개발 행동을
라우팅하는 형태로 구체화했다.

## 세 계층

1. **근거 계층** — 현재 코드, 테스트, `project.yml`, 제품 문서, 실행 결과,
   Git 이력. 에이전트는 이 계층을 임의로 복제해 `raw/`에 쌓지 않는다. Git이
   버전 고정 원본 역할을 한다.
2. **위키 계층** — `wiki/`의 요약, 구조, 플레이북, 결정. 에이전트가
   유지하고 사용자가 검토한다.
3. **운영 계층** — 루트 `AGENTS.md`와 이 문서. 언제 무엇을 읽고 수정하고
   검증할지 정의한다.

## 출처 우선순위

충돌이 있으면 아래 순서로 판단한다.

1. 재현 가능한 현재 실행 결과
2. 현재 코드·테스트·빌드 설정
3. Apple 등 공급자의 최신 공식 문서
4. 저장소의 제품·설치·QA 문서
5. 위키의 종합 페이지와 결정 기록
6. 오래된 작업 로그와 Git 커밋 메시지

상위 근거가 위키와 다르면 작업을 멈추지 말고 현재 사실을 확인한 뒤 위키를
수정한다. 과거 결정이 바뀌면 ADR을 삭제하지 않고 `superseded`로 표시하고
대체 결정을 링크한다.

## 페이지 종류

| 위치/종류 | 역할 | 수정 시점 |
| --- | --- | --- |
| `index.md` | 전체 탐색과 작업 라우팅 | 페이지 생성·이동·역할 변경 |
| `product.md` | 제품 의도와 UX 불변 조건 | 범위나 사용자 흐름 변경 |
| `architecture.md` | 현재 시스템의 종합 구조 | 경계·데이터 흐름·저장 방식 변경 |
| `tools.md` | 작업별 도구와 행동 규칙 | 더 안전하거나 재현 가능한 방식 발견 |
| `playbooks/` | 반복 가능한 작업·검증 절차 | 절차나 환경 제약 변경 |
| `decisions/` | 선택과 이유, 결과를 보존하는 ADR | 되돌리기 어려운 판단을 채택·교체 |
| `log.md` | 완료된 작업의 시간순 감사 기록 | 의미 있는 작업 검증 후 |

## 페이지 메타데이터

모든 위키 페이지는 다음 YAML frontmatter를 가진다.

```yaml
---
type: architecture
status: active
updated: "2026-07-19"
tags: [sessions, persistence]
sources:
  - Sources/DamSetCore/WorkoutEngine.swift
---
```

- `type`: `index`, `schema`, `product`, `architecture`, `tools`, `playbook`,
  `decision`, `log` 중 하나
- `status`: `active`, `draft`, `deprecated`, `superseded` 중 하나
- `updated`: 내용을 마지막으로 확인한 날짜
- `tags`: 탐색용 짧은 단어
- `sources`: 사실을 뒷받침하는 저장소 상대 경로. 인덱스·스키마·로그처럼
  운영 자체가 근거인 페이지는 생략할 수 있다.
- `last_verified`: 도구 버전·기기 상태처럼 쉽게 변하는 사실을 실제로
  재확인한 날짜. 필요한 페이지에만 둔다.
- 외부 정보가 핵심이면 본문 가까이에 공식 링크와 확인 날짜를 적는다.

내용을 수정할 때는 `updated`를 작업 날짜로 바꾸고, 환경 사실을 다시
확인했다면 `last_verified`도 함께 바꾼다.

## 새 페이지와 기존 페이지의 경계

- 다른 페이지에서 독립적으로 링크할 고유 개념·결정·반복 절차면 새로 만든다.
- 기존 개념의 속성, 구현 현황, 새로운 근거는 기존 페이지를 수정한다.
- 페이지를 만든 즉시 `index.md`에 한 줄 요약과 링크를 추가한다.
- 같은 개념의 유사 페이지를 만들기 전에 `rg -n "<term>" wiki`로 검색한다.

## 쓰기 기준

위키에 남길 내용:

- 여러 파일을 함께 봐야 이해되는 데이터 흐름과 불변 조건
- 특정 작업에서 어떤 도구·순서·검증이 필요한지
- 실제 실패를 막아 주는 제약과 재현 가능한 해결책
- 선택하지 않은 대안까지 알아야 이해되는 설계 결정
- 현재 해결되지 않았고 다음 작업의 범위를 바꾸는 위험

로그에만 남길 내용:

- 어떤 범위를 언제 바꾸고 무엇으로 검증했는지
- 그 작업에서 위키의 어느 페이지가 갱신됐는지

남기지 않을 내용:

- 채팅 전문, 사고 과정, 토큰/시간 사용량
- 비밀번호, 토큰, 개인 키, 기기 식별자 같은 비밀 또는 개인 정보
- 아무 판단도 바꾸지 않는 긴 파일 목록
- 재현되지 않았고 후속 가치가 없는 임시 오류

## 작업 흐름

### Ingest

새 문서, 새 플랫폼 제약, 중요한 실행 결과가 들어오면 관련 현재 근거를 읽고
기존 페이지에 통합한다. 요약 파일 하나만 추가하고 연결 페이지를 방치하지
않는다.

### Query

질문에 답할 때 인덱스에서 후보 페이지를 찾고 현재 근거로 확인한다. 답변에서
재사용할 분석이나 새 연결이 생기면 적절한 페이지에 되돌려 쓴다.

### Change

코드 변경은 [코드 변경 플레이북](playbooks/code-change.md)을 따른다. 구현과
동시에 불변 조건·도구·결정이 바뀌었는지 확인한다.

### Lint

`ruby Scripts/check_agent_wiki.rb`는 필수 파일, frontmatter, 상대 링크,
인덱스 누락, 로그 제목 형식을 검사한다. 주기적인 의미 검사는 코드와 페이지의
불일치, 중복 페이지, 고아 페이지, 서로 모순되는 결정, 오래된 `last_verified`
정보를 추가로 찾아야 한다.

## 로그 규칙

`log.md`는 오래된 항목에서 최신 항목 순으로 끝에 추가한다. 제목은
`## [YYYY-MM-DD] kind | title` 형식을 지켜 Unix 도구로 찾을 수 있게 한다.
항목은 작업의 결과와 검증을 재현할 만큼만 쓰고, 자세한 지식은 종합 페이지나
ADR에 둔다.
