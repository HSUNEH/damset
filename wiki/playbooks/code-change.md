# 코드 변경 플레이북

## 1. 범위 확정

- 요청의 관찰 가능한 완료 조건을 한두 문장으로 정리한다.
- `git status --short`로 기존 변경을 확인한다.
- [인덱스](../index.md)의 관련 페이지와 최근 로그를 읽는다.
- `rg`로 구현, 호출자, 테스트, 문서의 현재 상태를 함께 찾는다.

## 2. 위험 지도

변경이 아래 경계를 건드리는지 표시한다.

- 정규 세션 또는 상태 전이
- 저장 형식과 이전 데이터
- App Intent와 Live Activity
- 벽시계 휴식과 알림/오디오
- Xcode 타깃·서명·entitlement
- 운동 중 핵심 화면의 한 손 조작과 가시성

경계를 하나 이상 건드리면 [시스템 구조](../architecture.md)와
[검증 플레이북](verification.md)의 해당 행을 다시 확인한다.

## 3. 구현

- 규칙은 가능하면 `DamSetCore`에 두고 앱과 Intent가 같은 경로를 호출하게 한다.
- UI 표시를 위해 상태를 별도로 복제하지 않는다.
- 비동기 저장이 필요한 완료 동작은 저장 성공 전에 화면을 확정하지 않는다.
- 새 모델 필드는 이전 JSON에 없을 때의 디코딩 기본값과 모든 소비자를
  확인한다.
- 프로젝트 구조는 생성된 `.xcodeproj`보다 `project.yml`을 먼저 수정한다.
- 발견한 지속 지식이 기존 위키와 다르면 구현 중 해당 페이지를 바로 고친다.

## 4. 검증

- 가장 가까운 단위 테스트부터 실행하고 통과하면 넓은 게이트로 확장한다.
- UI·ActivityKit·기기 정책은 컴파일 성공만으로 완료 처리하지 않는다.
- 실패한 검증은 원인을 분류한다: 코드 회귀, 환경 부재, 권한/사람 관찰 필요.
- 환경 때문에 실행하지 못한 단계는 실행한 것처럼 기록하지 않고 남은 위험을
  정확히 쓴다.

## 5. 지식 통합과 인계

- 구조·도구·불변 조건·결정이 바뀌었으면 관련 종합 페이지 또는 ADR을 갱신한다.
- 새 페이지는 인덱스에 연결한다.
- `ruby Scripts/check_agent_wiki.rb`와 `git diff --check`를 실행한다.
- `wiki/log.md` 끝에 작업, 중요한 행동, 결정, 실제 검증 결과를 한 항목으로
  기록한다.
- 마지막 diff에서 코드와 문서의 용어·현재 상태가 일치하는지 확인한다.
